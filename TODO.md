Original plan below kept as a marker for what was intended; behaviour matches.

Today `PhonoContext::scan_library` is a single loop: per file → TOC → sidecar enrich → upsert rip → identify (network round trips), with `LibraryChanged` only sent at the very end. The user-visible result: a folder of N CUEs takes N × (1 sec MB + provider chatter) before *any* row appears in the album-list, and the GUI provides no per-row affordance for "I'm waiting on the network for this one."

Goal: split the scan into a **fast metadata phase** (walk + TOC + sidecar + upsert) that streams `LibraryChanged` per row, and a **slow identification phase** that runs as a serialized worker queue (provider rate limits make parallelism wrong here) with a Status column on the album list reflecting per-row state.

**Data model**
- `rip_files.identification_state` enum column (TEXT): `unscanned` / `queued` / `working` / `identified` / `unidentified` / `failed`. Defaults to `unscanned`. Replace the implicit "disc_id IS NULL → unidentified" inference with this explicit state — `Unidentified` becomes "tried and nothing matched", `Queued` becomes "row exists but identify hasn't run yet". Pre-release schema bump (no migration).
- Optional `last_state_change_at` RFC3339 for sorting + UI staleness checks.

**Library API**
- Split `phono-junk-lib::scan::ingest_path` into two functions:
  - `ingest_metadata(conn, path, opts) -> IngestOutcome` — TOC read, sidecar collect/enrich, `RipFile` upsert (state set to `Queued` when identification is wanted, `Unscanned` otherwise). No network. Always emits `ScanEvent::Ingested { rip_file_id }` so the GUI can stream a `LibraryChanged` between every row.
  - `identify_one(ctx, conn, rip_file_id, force_refresh) -> IdentifiedDisc` — pulls the RipFile + TOC + DiscIds from the row, runs the existing `identify_disc` pipeline, transitions state to `Identified` / `Unidentified` / `Failed`.
- `scan_library` becomes the metadata-only walker; identification work is dispatched separately by callers (CLI loops it inline, GUI queues it on a worker).
- Sidecar refresh on cache hit (Bug 2 above) lives in `ingest_metadata` — both phases share the same metadata-write path.

**GUI worker**
- New `backend/identify_queue.rs` owning a single `mpsc::Receiver<RipFileId>` worker thread (serialized — provider rate-limit constraint). Scan dispatcher pushes ids onto it as `ingest_metadata` returns; GUI bulk Identify pushes too. Channel emits `RowStateChanged { rip_file_id, state }` messages so the table re-renders the affected row only.
- `BackgroundOperation` for the queue shows count remaining, not per-file progress.

**Album list view**
- New leading Status column in `views/album_list.rs::table` rendering the state with an icon: `unscanned` blank, `queued` `…`, `working` an `egui::Spinner` (auto-repaints), `identified` `✓`, `unidentified` `⚠`, `failed` `✕`. Single source of truth — `RipFile.identification_state`. Gated to redraw via `ctx.request_repaint_after(100ms)` when any row is `working`.
- `ListRow` / `UnidentifiedRow` (or a new envelope) carry the state through `load_list_entries`.

**CLI parity**
- `phono-junk-cli scan` runs `ingest_metadata` then loops `identify_one` over the queued ids. Same two-phase shape — no behavioral divergence from GUI. Existing `scan --no-identify` still works (skips phase 2). Add `phono-junk-cli identify --queued` to drain the queue without re-walking the tree.

**Tests**
- `scan_library` returns immediately after metadata phase; rows appear with `identification_state = Queued`.
- `identify_one` transitions a queued row to `Identified` / `Unidentified` / `Failed` and persists `last_state_change_at`.
- Re-running `identify_one` on a `Failed` row retries (state-not-cache).
- GUI test: `RowStateChanged` only redraws the affected row (mock the message channel).

**DRY / best practices**
- `ingest_metadata` and `identify_one` are the single shared primitives — CLI loops them, GUI queues them, no two implementations of "scan one CUE."
- The new state column subsumes the implicit "disc_id IS NULL → unidentified" check; `crud::list_unidentified_rip_files` becomes `WHERE identification_state = 'unidentified'` instead of a NULL check, removing an ambiguity.
- `BackgroundOperation` already exists from Sprint 15 — the queue is a new dispatcher, not a new widget.

**Cross-repo note**
- The serialized-worker-queue dispatcher (one rate-limited consumer per resource pool) is a candidate for `junk-libs-gui` once retro-junk grows a similar "scrape one ROM at a time" workflow. Document but don't extract on the first consumer.

## Open questions

- Which Japanese-focused disc database should follow Tower MDB? Remaining candidates: VGMdb (strong on anime / game OSTs, public JSON-ish endpoints), CDJapan (commercial catalog, format unclear), HMV Japan (retailer, likely scrape-only). Tower MDB landed first because it proved scrapeable with clean semantic HTML and 4.4M releases keyed on barcode / catalog number.
- **Sprint 13 `list --filter` vocabulary on `genre` / `language`.** Resolved Sprint 13: MVP ships `--artist`, `--year`, `--country`, `--label` (all schema-backed on `Album` / `Release`) and drops `genre` / `language`. Reopening them requires two pieces: a schema column or associative table, and a provider enrichment pass (MB exposes genre via release-group tags and language via work-level MBID links — neither fits as a simple column). When both land, the flags slot into the existing `ListFilters` struct at `phono-junk-lib/src/list.rs`.
- How to model HDCD / SACD / mixed-mode carriers — extend `Disc` with a carrier-type enum, or add to `extra` map?
- User-facing vs. archival override semantics — should overrides fully replace scraped data, or sit alongside it as "preferred display" while keeping the provider's value for audit?
